# EURUSD Trading Simulator — Implementation Plan (Agentic AI Coder, Flexible)

## Starting Point (Already Available Artifacts)
The user has exported and downloaded **two trained models** from Colab:

- `vol_model_elasticnet.joblib`  
  - A **scikit-learn Pipeline** (ElasticNet + scaler inside the pipeline if used).
  - Purpose: predict next-hour volatility proxy (e.g., `y_vol_1h`).

- `dir_model_lightgbm.joblib`  
  - A **LightGBM classifier** trained for next-hour **direction**.
  - Purpose: output `P(up | X_t)` via `predict_proba`.

The user will also prepare a local dataset file compatible with these models.

> Agent is allowed to adapt implementation details as needed, but must respect time-series constraints and feature alignment.

## Project Goal
Build a local prototype that:
- Loads the two `.joblib` models.
- Loads a prepared dataset (last year or any chosen range).
- Runs an interactive **simulation** where the user chooses:
  - start “current time”
  - starting capital
  - strategy type
  - simulation horizon (e.g., 3 hours)
- Outputs whether the strategy won/lost money + timeline + equity curve.

This prototype is a **simulation/backtest demo**, not live trading.

- `datetime` (hourly timestamps; unique; sorted)
- `ret_fwd` (forward 1-hour return aligned to each row; used for P&L update)
- Feature columns used at training time (must match expected input of both models)

Recommended:
- Provide a `feature_list.json` (ordered list of feature columns) to avoid column mismatch.

## High-Level Architecture (Agent can customize)
Separate responsibilities into independent modules/services (naming flexible):

1) **Data Layer**
- Load dataset and validate index continuity
- Provide access to feature matrix `X_t` at time `t`
- Provide `ret_fwd` at time `t`

2) **Model Layer**
- Load the two models (`joblib.load`)
- Provide:
  - `predict_vol(X_t) -> sigma_hat`
  - `predict_direction_proba(X_t) -> p_up`

3) **Strategy Layer**
Implements trading decisions using model outputs.

4) **Simulation Engine**
- Runs through the selected time window step-by-step
- Applies decisions + costs
- Produces equity curve + performance metrics + timeline

5) **Interface Layer (later)**
- CLI initially or minimal local API; frontend can be added afterward.

## Trading Strategies (Explainable and Reusable)

### Strategy 1 — Always-on Direction Strategy
**Purpose:** baseline “always in the market” strategy to test whether directional signal accumulates.

At each hour `t`:
- Compute `p_up = P(up | X_t)`
- Decision:
  - if `p_up > 0.5` → **LONG**
  - else → **SHORT**
- Profit update:
  - `strategy_ret = position * ret_fwd`

Characteristics:
- High trading frequency
- Sensitive to transaction costs
- Good for demonstrating model-driven decisions

---

### Strategy 2 — Selective Confidence Strategy (Recommended)
**Purpose:** trade only when the model is confident to increase signal quality and reduce noise.

At each hour `t`:
- Compute `p_up = P(up | X_t)`
- Decision:
  - if `p_up > upper` → **LONG**
  - if `p_up < lower` → **SHORT**
  - else → **FLAT** (no trade)

Optional extension (recommended): **volatility filter**
- Compute `sigma_hat` from the volatility model
- Only allow LONG/SHORT if `sigma_hat > vol_threshold`
- Otherwise force FLAT

Characteristics:
- Lower trade frequency
- More realistic for weak FX direction signal
- Naturally integrates both models (direction + volatility)

---

## Simulation Logic (Minimal Requirements)
Simulation must:
- Start with **normalized** or real capital (e.g., 1000)
- For each simulated hour:
  1) extract feature vector `X_t`
  2) compute model output(s): `p_up` (and optionally `sigma_hat`)
  3) compute position based on chosen strategy
  4) compute step return: `position * ret_fwd`
  5) apply cost model if enabled
  6) update equity: `capital *= (1 + step_return)`
  7) log timeline row

---

## Transaction Costs (Keep Simple)
Include a toggle for a simple cost proxy:
- Charge a fixed cost when position changes:
  - `abs(pos_t - pos_{t-1}) * cost_per_change`
This allows:
- Long ↔ Short flips to cost more than staying in same direction
- Flat ↔ Long/Short to cost once

## Outputs (What the Prototype Must Produce)
At the end of each simulation run:
- Summary:
  - final capital
  - profit/loss
  - percent return
  - number of trades
  - coverage (% of hours with non-zero position)
- Equity curve series (timestamp, equity)
- Timeline log (for each hour):
  - datetime
  - p_up
  - sigma_hat (if used)
  - position
  - ret_fwd
  - cost
  - step return
  - capital after update

Output format should be easily consumable later by a UI:
- JSON + optional CSV exports.
